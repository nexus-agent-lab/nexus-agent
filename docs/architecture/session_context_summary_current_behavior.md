# Session Context Summary Current Behavior

## Purpose

This note records how conversation history compaction and `PREVIOUS CONTEXT SUMMARY` currently work, so the team can return later to improve the design without re-deriving the existing behavior from code.

## Relevant Files

- `app/core/session.py`
- `app/core/chat_session_bootstrap.py`
- `app/core/agent.py`
- `app/core/worker.py`

## Current Model

The current chat-context strategy is a two-tier model:

1. older history is compacted into LLM-generated summaries
2. recent history is kept as raw messages

When a new turn starts, the backend reconstructs prompt context from:

1. the stored historical summary, if one exists
2. the most recent raw messages
3. the current incoming user message

## When `PREVIOUS CONTEXT SUMMARY` Appears

`PREVIOUS CONTEXT SUMMARY` is injected only when the current session already has at least one stored `SessionSummary`.

The injection happens in `app/core/chat_session_bootstrap.py`:

- resolve or create the session
- call `SessionManager.get_history_with_summary(session.id, limit=10)`
- if `history_summary` is non-empty, prepend a `SystemMessage`:
  - `## PREVIOUS CONTEXT SUMMARY`
- append recent raw messages
- append the current user message

So this summary is not shown or injected on every turn. It appears only after compaction has already happened at least once.

## When Summary Generation Is Triggered

Summary generation is handled in `SessionManager.maybe_compact(...)` inside `app/core/session.py`.

Current defaults:

- compaction threshold: `20` unarchived messages
- keep raw after compaction: last `15` messages
- restore raw during next turn bootstrap: last `10` messages

Trigger behavior:

- after a persisted assistant or tool message, `SessionManager.maybe_compact(session_id)` may run
- if unarchived message count is `> 20`, `compact_session(...)` is called
- `compact_session(...)` archives older messages and leaves the newest `15` raw

## How Summary Text Is Produced

The summary is generated by the LLM, not by deterministic rules.

Inside `compact_session(...)`:

- older messages selected for archiving are formatted into a text block
- each message is labeled by role/type
- that block is sent to the LLM with a prompt asking for:
  - a concise, factual paragraph
  - key decisions
  - important facts
  - user preferences
  - outcomes
  - no focus on greetings or small talk

The generated result is saved in `SessionSummary.summary`.

## What Content Is Included In Summary Source Material

The source material for summary generation can include:

- user messages
- assistant messages
- tool messages

This means the stored summary may reflect:

- what the user asked for
- what the assistant answered
- what tools found or returned

It is not a raw replay. It is an LLM-written abstraction of archived turns.

## What Content Is Sent To The Main LLM On A New Turn

Today the reconstructed context can include all of the following:

1. `PREVIOUS CONTEXT SUMMARY` system message, if any summary exists
2. recent raw human messages
3. recent raw assistant messages
4. recent raw tool messages
5. the current user message

So yes: both user and assistant history may be sent again on later turns, and tool outputs may also be represented.

## Limits And Guardrails

### Raw history restore limit

The bootstrap path currently restores up to `10` recent raw messages.

This is a message-count limit, not a strict token-budget limit.

### Post-compaction retained raw history

After compaction, the last `15` unarchived messages remain raw in storage.

### Tool output pruning

Large tool outputs are pruned before storage when they exceed `2000` characters.

This reduces the chance that raw recent context is overwhelmed by a giant tool result.

### Summary accumulation

`get_history_with_summary(...)` currently concatenates all stored summaries together.

There is no second-level summary-of-summaries pass yet.

## Current Tradeoffs

### Strengths

- follow-up turns like `继续呀` can still see recent raw wording
- older context does not grow without bound
- tool-heavy conversations are partially protected by output pruning
- the client does not need to replay full history

### Weaknesses

- summary quality depends on the LLM
- important details may be lost during summarization
- limits are based on message counts and character thresholds rather than a real token budget
- all stored summaries are concatenated directly, which may become inefficient over long sessions
- tool outputs may still influence the summary in ways that are noisier than ideal

## Suggested Future Optimization Areas

When revisiting this subsystem, likely improvements include:

- replace message-count restore limits with token-budget-based trimming
- use a structured summary schema instead of one freeform paragraph
- distinguish user intent, decisions, completed work, and pending next steps
- add summary-of-summaries compaction for very long sessions
- reduce noisy tool-output contribution to long-term summary
- add special follow-up handling for short references such as:
  - `继续呀`
  - `然后呢`
  - `按刚才那个`
  - `接着做`

## Current Recommendation

Keep the present design for now, because it is sufficient to restore basic multi-turn continuity after the recent `thread_id` fix.

Return later to optimize only when:

- sessions become substantially longer
- continuity quality degrades in practice
- prompt costs become a concern
- the team wants stronger guarantees around preserving precise facts
